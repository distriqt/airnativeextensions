algorithmproperty
public var algorithm:String = native

The algorithm represents the method we use to calculate the position and motion of the device.

This may use a combination of sensors in a 'fusion' or just raw sensor data. See the individual ALGORITHM_ definitions for more details on each algorithm.

formatproperty 
public var format:String = quaternion

This controls what data the extension returns when device motion is detected and the type of update events that are fired from the extension.

You can select an older style euler angles (though we highly suggest you do not use these) or something more reliable such as quaternions

rateproperty 
public var rate:int = 3

The rate at which updates should be requested from the sensors. Note that this rate is the query rate of each individual sensor.

See the SensorRate.SENSOR_DELAY_ values. Default is SensorRate.SENSOR_DELAY_NORMAL

ALGORITHM_FUSIONConstant
public static const ALGORITHM_FUSION:String = fusion

This algorithm uses a complimentary filter sensor fusion algorithm. This is a custom written algorithm.

This algorithm is under experimentation, we advise you to test throughly to see if it meets your applications needs.

ALGORITHM_NATIVEConstant 
public static const ALGORITHM_NATIVE:String = native

This is the default and suggested algorithm for use in most common cases. It uses the underlying native systems implementation of a sensor fusion algorithm and converts this data into the requested return format.

FORMAT_EULERConstant 
public static const FORMAT_EULER:String = euler

The simplest and most problematic return values. Only use these if you really cannot use one of the other formats.

You can encounter odd motion in the form of a Gimbal lock using these values where you'll lose a degree of freedom. http://en.wikipedia.org/wiki/Gimbal_lock

FORMAT_QUATERNIONConstant 
public static const FORMAT_QUATERNION:String = quaternion

This is the best format to retrieve the device rotation information.

Quaternions are a number system extending complex numbers, and are sort of similar to a vector in concept in three dimensions. They provide a very convenient way to represent orientations and rotations in three dimensions.

Read more on quaternions here:

  • http://en.wikipedia.org/wiki/Quaternions_and_spatial_rotation
  • http://content.gpwiki.org/index.php/OpenGL%3aTutorials%3aUsing_Quaternions_to_represent_rotation
