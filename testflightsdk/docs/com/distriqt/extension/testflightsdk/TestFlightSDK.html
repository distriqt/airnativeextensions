This class is the Test Flight SDK Native Extension interface. It implements the iOS-based TestFlight SDK for debugging and feedback services through the TestFlight platform.

This Native Extension requires the additional use of the TestFlight service, available from http://testflightapp.com.

The extension currently implements the following versions of TestFlight SDK:

  • iOS: 3.0.0
  • Android: 1.3
Related Links for the TestFlight Service
  • TestFlight SDK Documentation: https://testflightapp.com/sdk/doc/

You must implement the TestFlight service for your application. Guides and tutorials for general setup and SDK functionality are available from the TestFlight website.

iOS

Notes and Limitations

The UDID requirement of TestFlight has been removed so the extension will be fine for usage in release applications as well as the previous beta usage.

Please note that for this functionality to work properly with the TestFlight system, you must upload and distribute application builds through TestFlight. Checkpoints and logging etc will register with TestFlight through a normal debug or ad-hoc build, however for full reporting statistics and user feedback etc to be displayed effectively, you must distribute your build through the TestFlight build system.

Android

You'll need to make sure you have the following lines in the manifest additions node of your application descriptor file. 

	 <uses-permission android:name="android.permission.INTERNET"/>
	 <uses-permission android:name="android.permission.ACCESS_NETWORK_STATE"/>
Android Debug Certificate & Application Signing

Reference: http://developer.android.com/tools/publishing/app-signing.html

In order for this ANE to work on Android in conjunction with the TestFlight website you must sign your application with the Android debug keystore and not the Flash Builder debug certificate or your Play store release certificate. Without this step there will be nothing reported to your TestFlight dashboard, however you should still receive information through to FlightPath if you have access to this.

When packaging the application you need to use the android debug certificate. Firstly convert the debug certificate to a pkcs12 format from the JKS format it's currently in. The file is normally located at ~/.android/debug.keystore. Run the following in the .android directory to create this p12:

keytool -importkeystore -srckeystore debug.keystore -srcstoretype JKS -deststoretype PKCS12 -destkeystore debug.p12

Once the p12 file has been created, change the certificate in the Android packaging settings to point to this debug.p12 file.

Note: If the debug.keystore doesn't exist try creating and building an empty android application in eclipse using the Android Application project type.

Finally when uploading the application to TestFlight you may receive a message saying that your app token was not found in your application. This error will occur even if you have correctly added your app token so you can ignore this error as long as you have provided the correct app token to the startTestFlight call.

public function log(message:String):Boolean

Calls the log / TFLog method of the native TestFlight SDK. This is a generic logging system used by TestFlight.

Logs appear in the session details for beta/debug versions only. They may take a while to appear and generally are only updated when a session ends. i.e. the user restarts the application or the session timeout occurs.

startTestFlight()method 
public function startTestFlight(appToken:String, debugMode:Boolean = true):Boolean

Calls the takeOff method of the native TestFlight SDK and begins the TestFlight session.

You'll need to get your Application Token which you can find at http://testflightapp.com/dashboard/applications/ . Select the application you are using from the list choose the SDK option and the application token for this application will be there.

Parameters

appToken:String — Your application token.
 
debugMode:Boolean (default = true) — Whether to utilise the iOS UDID as part of the initialisation. Defaults to true. This is required to be true in order for some TestFlight features to function correctly. This option should be set to false for a production/release version of the application, but generally always set to true when debugging or testing your app during development.

Returns
Boolean — true If the initialisation method completed.

Examples
This simple example shows a very basic take off, log and checkpoint. 
     
     try
     {
         TestFlightSDK.init( DEV_KEY );
         
         // Call startTestFlight with the appToken for the correct platform 
         if (TestFlightSDK.service.version.indexOf( "Android" ) != -1)
         {
             TestFlightSDK.service.startTestFlight( TESTFLIGHT_APPTOKEN_ANDROID );
         }
         else
         {
             TestFlightSDK.service.startTestFlight( TESTFLIGHT_APPTOKEN_IOS );
         }
     
         TestFlightSDK.service.log( "Testing a log message..." );
         TestFlightSDK.service.passCheckpoint( "TEST_CHECKPOINT_1" );
         
     }
